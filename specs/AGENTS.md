# Specification Guidelines

This subtree is the documentation source of truth for implementation.

Rules:
- keep requirements, technical scope, constraints, acceptance criteria, and validation strategy explicit
- update the relevant spec or matrix before or alongside code changes when the repo uses spec-first governance
- record unresolved questions instead of silently guessing
- verify version-sensitive technical guidance against official n8n docs before freezing it into specs
- keep implementation plans modular and testable

A spec is not complete until another engineer could implement the core behavior without guessing.
